# Ghidra extraction notes

Observations about what Ghidra's headless extraction captures and
what it legitimately omits, calibrated against ELF symbol table
ground truth from `arm-none-eabi-nm`. Updated whenever the
`notes/queries/coverage.sql` coverage report surfaces a new class of
discrepancy.

## The 84-vs-96 "gap" on Pico blinky (2026-04-04)

Initial coverage run on `targets/pico_blinky/blink.elf`:

| metric                     |  value |
|----------------------------|-------:|
| nm text symbols (T + t)    |    121 |
| unique nm addresses        |     96 |
| Ghidra functions           |     84 |
| agreement by address       |     66 |
| only in nm                 |     30 |
| only in Ghidra             |     18 |
| raw coverage               |  68.8% |

The raw coverage number looks alarming. It isn't. Every missing
address is explained, and the underlying categories are all cases
where Ghidra is correct.

### Why 121 nm symbols collapse to 96 unique addresses

25 of the 121 nm symbols are **aliases at shared addresses**:
zero-length labels generated by the Pico SDK linker script for
section boundaries and dispatch tables. Examples at a single
address:

```
0x100001e8  __binary_info_header_end
0x100001e8  __embedded_block_end
0x100001e8  _entry_point
```

These are not three functions; they are three names pointing at the
same byte. Deduplicating gives 96 real addresses.

### Why 30 of the 96 unique addresses are missing from Ghidra

Breakdown of the 30 addresses nm reports but Ghidra does not treat
as functions:

1. **Section and region boundary markers** (~16 addresses).
   `__boot2_start__`, `__boot2_end__`, `__VECTOR_TABLE`, `__vectors`,
   `__flash_binary_start`, `__default_isrs_end`, `__data_end__`,
   `__TMC_END__`, `__mutex_array_start`, `__mutex_array_end`, etc.
   All zero-size, all linker-emitted labels. Not function bodies.
   Ghidra correctly does not create functions for them.

2. **Function pointer table entries in the `pre_init` array**
   (11 addresses, all named `__pre_init_*`, all 4 bytes).
   `__pre_init_runtime_init_clocks`, `__pre_init_runtime_init_mutex`,
   etc. These are 4-byte slots in a table that the Pico SDK runtime
   walks on boot to call each entry. The slots contain function
   *pointers*, not function *bodies*. Each pointer targets a real
   function that Ghidra **does** have: the
   `notes/queries/coverage.sql` has a companion check proving every
   `__pre_init_foo` points at a Ghidra-extracted `foo` function.
   Ghidra correctly treats the pre_init table as data, not code.

3. **RAM-resident data objects mislabeled as text** (3 addresses).
   `striped_spin_lock_num` (1 byte), `default_alarm_pool` (24 bytes),
   `print_mutex` (8 bytes). All at RAM addresses (0x2000xxxx). These
   are declared with nm type `t` because of a Pico SDK linker-script
   quirk in how statically-initialized RAM data is attributed to
   sections with text-like flags, but they are data structures, not
   code. Ghidra correctly does not create functions for them.

**None of the 30 represents a real extractor miss.** The
inner-loop query at the bottom of `coverage.sql` filters nm to
"has size AND in flash AND not a `__pre_init_*` pointer," which
collapses the 96 unique addresses to ~46 real flash function
bodies, and Ghidra finds **all 46**.

### Why 18 Ghidra functions are missing from nm

The reverse direction. Ghidra's extraction finds 18 function bodies
that have no nm symbol. Categories:

1. **Weak default ISR handlers** (6 addresses, 2 bytes each).
   `isr_invalid`, `isr_nmi`, `isr_hardfault`, `isr_svcall`,
   `isr_pendsv`, `isr_systick`. Each is a single `b .` (infinite
   loop). The Pico SDK declares these weak so user code can override
   them, and the linker appears to strip the symbol entries when
   they remain at their weak definitions — but the code bodies
   survive in the vector table targets, and Ghidra recovers them
   from the vector table plus DWARF debug info.

2. **`runtime_init_*` boot callbacks** (10 addresses, 8–184 bytes).
   Real Pico SDK runtime initialization functions referenced from
   the `pre_init` table above. Their symbols are apparently
   localized or elided in the final link (possibly because they're
   declared `static inline` but emitted out-of-line due to address-
   taken usage in the pre_init table), but Ghidra recovers them from
   DWARF.

3. **`_exit`** (1 address, 4 bytes). newlib's tiny `_exit` wrapper;
   almost certainly a local symbol the linker stripped.

4. **`hard_assertion_failure`** (1 address, 8 bytes). Weak or
   hidden symbol; Ghidra recovers it from DWARF.

These 18 functions are Ghidra being **more** complete than nm, not
less. They all have sizes, names (from DWARF), and callable code
bodies.

## Takeaway for the extractor

Ghidra's `functionManager.getFunctions(True)` iterator is the
correct source of "what is a function" for the pipeline's purposes.
nm's T/t symbol set is a noisy overestimate that includes:

- Data labels with text-like section flags
- Section boundary markers
- Function pointer table entries
- Aliases at shared addresses

and simultaneously misses:

- Weak default handlers the linker strips
- `static inline` functions forced out-of-line by address-taking
- Some newlib and runtime internals

For regression purposes, the `notes/queries/coverage.sql` report is
useful not because the raw percentage is meaningful but because
**the specific set of missing addresses should stay stable** across
pipeline changes. If a future extractor change causes a
previously-explained missing address to reappear on the gap list, or
a previously-extracted Ghidra function to disappear, that is a
regression worth investigating.

## Debug-section strings in export_strings.py (2026-04-04)

On the first Step 3 run, `export_strings.py` emitted **334 strings**
for Pico blinky, of which only **9 were in loaded flash**
(0x10000000+). The other 325 were DWARF `.debug_str` / `.comment`
entries that Ghidra loads into overlay address spaces at very low
addresses (0x0 to ~0x1950) — toolchain metadata, compilation unit
names, source paths, the GCC banner, local variable names, etc.

Fix: `export_strings.py` now filters via
`program.getMemory().getLoadedAndInitializedAddressSet().contains(addr)`
before checking `StringDataInstance.isString`. Only strings that
live in the binary's runtime memory image get emitted.

This filter is the right default for every extractor that walks
listing data program-wide. `export_functions.py` and
`export_basic_blocks.py` are already implicitly safe because they
iterate FunctionManager and BasicBlockModel respectively, both of
which only return results from loaded code. `export_xrefs.py` and
`export_calls.py` iterate from within function bodies, which are
also loaded by construction. Only program-wide data iteration
(strings, and any future extractor that walks `getDefinedData` or
the full Listing) needs the explicit filter.

If a future target shows a suspicious jump in strings row count,
the first check is whether the new strings are in loaded memory or
in debug overlays.

## Zephyr hello_world on qemu_cortex_m3 (2026-04-05)

Second target added. Coverage is dramatically cleaner than Pico:

| metric                     | pico_blinky | zephyr_hello_world |
|----------------------------|-----------:|-------------------:|
| nm unique addresses        |         96 |                100 |
| Ghidra functions           |         84 |                110 |
| agreement by address       |         66 |                 97 |
| only in nm                 |         30 |                  3 |
| only in Ghidra             |         18 |                 13 |
| coverage %                 |       68.8 |               97.0 |

The 3 "only in nm" unique addresses on Zephyr (6 raw rows including
aliases) are the same categories we saw on Pico, smaller in number:

- Section/region boundary markers: `__rodata_region_start`,
  `__text_region_end`, `__rom_start_address`, `_vector_start`,
  `_vector_table` — zero-size labels at link-script boundaries.
- `_irq_vector_table` (172 bytes) at 0x40 — the ARM Cortex-M
  interrupt vector table. It's a data structure of function
  pointers, not a function body, so Ghidra correctly doesn't create
  a Function object for it. The individual handler functions it
  points at *are* in the Ghidra extraction; the table itself is
  metadata about where they live.

So the 97% coverage number is real: Zephyr's picolibc-linked
hello_world has essentially no non-function text symbols, so the
raw nm vs Ghidra count comparison is much closer to meaningful on
this target than it was on Pico.

The 13 "only in Ghidra" functions on Zephyr are the same class of
finding as on Pico — weak symbols the linker stripped, inline
functions forced out-of-line, etc. Ghidra recovers them from DWARF.

**Takeaway across both targets:** the extractor's behavior is
identical; the raw coverage percentage difference is entirely a
property of the target's symbol table discipline, not the
extractor. Pico SDK ships with a lot of link-script labels and
function-pointer tables declared in `.text` sections; Zephyr
doesn't. For regression purposes, the useful invariant is still
"the specific set of missing addresses should stay stable between
pipeline runs for a given target," and that holds.

## Next targets

Future targets (FreeRTOS, Arduino AVR, ESP32 RISC-V) should rerun
the coverage query via `scripts/query < notes/queries/coverage.sql`
and add a short section here for any new category of missing
symbol that turns up. The categories we've seen so far are:

- Section boundary markers (present on every link-script-based
  ELF; Ghidra correctly omits them)
- Function pointer tables (Pico's pre_init, Zephyr's IRQ vector
  table — different shapes, same non-function-ness)
- Data objects in text-flagged sections (Pico SDK quirk)
- RAM-resident symbols marked T because of Pico linker script
  (may or may not appear on other targets)
- Weak-default handlers the linker strips (appear as "only in
  Ghidra" because DWARF preserves the names even when the symbol
  table doesn't)
